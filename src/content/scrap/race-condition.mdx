# レースコンディションとは？

レースコンディションとは、複数のスレッドやプロセルが同じデータに同時にアクセスし、実行されるタイミングや順序によって処理結果が変わってしまう問題のことです。
データベース更新トランザクションの競合もこのレースコンディションに該当するものです。

# なぜレースコンディションが起きるのか？
多くの処理はコンピューター内部で以下の3ステップで行われます。
1. READ
2. Modify
3. Write
ReadからWriteまでの間に別の処理が割り込んでデータを書き換えてしまうとこの処理が狂ってしまいます。

# レースコンディションを防ぐ方法
誰かがデータを使っている間は、他の人を待たせるという排他制御が必要になります。
## ロック (Lock/Mutex)
誰かがデータを使っている際は、鍵をかけて処理が終わるまで他のスレッドが入れないようにします
## アトミック操作 (Atomic Operations)
「読み込み」「修正」「書き込み」を分割負荷王な1つの動作として実行します。
## DBトランザクション
データベースレベルで一連の処理をひとまとめにして、整合性を保証します。

## レースコンディションの特徴
厄介さは、「時々しか起きない」つまりデバッグの再現度が低いことです。
デバッグしようにも、ログ出力処理のせいでタイミングずれて問題を発見できないことがあります。(ハイゼンバグ)
